### Fixed
- **Gravity hero attack sound spam**: throttled `attack_gravity` SFX to minimum 250 ms between plays, matching Air hero behavior.
- **Online: trust client prediction during active movement** — the previous reconciliation loop pulled the local player toward the server's authoritative position every frame, which fought intentional motion. Server simulation runs at 30 Hz (one physics step per tick) while the client predicts at 60 fps, so the client is structurally ~1 frame ahead of the latest server snapshot during continuous motion; reconciling produced a constant backward drag while running and a hard yank-back at the end of every dash. Reconciliation is now skipped entirely while `moveInput` is non-zero (or `isDashing` / dash grace is active). Convergence only runs while the player is idle (gentle 10 %/frame pull) or when the divergence is huge (>300 px → instant snap for teleports / death / extreme lag spikes). Net effect: full client-speed running and dashes, no more rubber-banding, while still correcting genuine mispredictions when the player isn't actively driving motion.
- **Online: projectiles dying mid-flight before reaching enemies** — when the server detected a projectile hit it removed the projectile from the next snapshot, and the client immediately dropped it from `projectiles[]`. Combined with the 100 ms interpolation delay this made projectiles visually pop out of existence before they reached the enemy on screen, especially noticeable with the auto-aim pickup. Fixed by carrying orphaned ghost projectiles forward across snapshot rebuilds: when a projectile id disappears from a new snapshot it's marked `_orphanAt` and kept in the array; the render loop interpolates it to its last buffered position, then drops it once `renderTime` passes that point. Projectiles now visibly complete their flight to the impact point and trigger the enemy hit-flash at the correct visual moment.
- **Online: internet-grade smoothness pass** — second wave of fixes after LAN testing still showed visible jumping. (1) Server tick rate bumped 20 Hz → 30 Hz (`TICK_MS = 33` in `server/simulation/constants.js`); smaller gaps between snapshots reduce coarseness of interpolation. (2) Player/enemy/projectile positions in the snapshot now use 1-decimal precision (`Math.round(x * 10) / 10`) instead of integer pixels — eliminates the integer-pixel quantization stutter at slow speeds without bloating bandwidth. (3) Snapshot interpolation buffer is now indexed by **server timestamp `s.t`** instead of client receipt time. Internet jitter caused 50 ms server intervals to stretch/compress to 30–80 ms in receipt time, making interpolation rate vary visibly; using server time keeps interpolation cadence in sync with actual world motion. (4) New `_onlineClockOffset` tracker locks to the minimum observed `clientNow − serverT` (least-delayed packet) and slowly drifts to follow clock skew; render time = `clientNow − offset − 100 ms` so renderTime advances smoothly each frame even between snapshots. (5) `_INTERP_DELAY_MS` raised 80 → 100 ms and snapshot ring buffer raised 4 → 6 entries to absorb wider internet jitter without falling back to extrapolation. (6) Own-player reconciliation moved from per-snapshot 8 % blend (which produced a visible 20 Hz wobble) to per-frame continuous pull at 6 %/frame (smooth at 60 fps); hard-snap threshold raised 80 px → 300 px so brief network hiccups, dashes, and knockbacks don't teleport the player. (7) Reconciliation suspended while `player.isDashing` and for an 18-frame grace window afterwards — a dash covers ~160 px in 10 frames and races a full server tick ahead, so reconciling mid-dash visibly yanked the player back to the server's lagging position. (8) `_onlineClockOffset` resets in `_resetGameState()` so a new session locks to a fresh server clock.
- **Online: smooth multiplayer rendering** — replaced direct snapshot-position application with a ring-buffer interpolation system (`_onlineInterpBuf`). Remote player, enemies, and projectiles are now rendered 100 ms behind the latest server time, interpolating between two buffered snapshots instead of jumping to the latest server position every tick. New ghost entities and single-snapshot startup still hard-snap as before; fallback extrapolation keeps rendering smooth while the buffer fills. Ghost enemies now show a brief white hit-flash when their HP drops between snapshots, giving immediate tactile feedback for projectile impacts without waiting for the server death event.
- **Online: Waker of Winds challenge system** — Air Hero wave-objective mechanic now works in online co-op. Server-side: `_checkWaveAdvance()` respects `world.objectiveLocked` set by `AirHero.update()`; `showNotification` is stubbed on the server world to relay objective alerts to both clients as `notification` events. Client-side: objective state is included in each player snapshot and applied on the guest so the HUD renders the correct progress; the guest skips local objective generation and boss-blocking (server is authoritative). Both host and guest see the same objective text and receive notifications for goal start, completion, and boss lock/unlock.
- **P2 HUD stuck on screen**: the bottom-right co-op HUD now hides immediately when a game ends (via `gameOver()` or `_resetGameState()`). Previously it remained visible after online co-op or local co-op games ended.
- **Online hero selection**: changing hero in the online lobby/pre-game screen now correctly applies to the local player's own character in-game. Previously `selectedHeroType` was not updated before `startGame()`, so the client still spawned the main-menu hero.
- **Online versus mode**: selecting versus mode now correctly suppresses enemy spawning and wave advancement on the server. Server simulation now handles PvP projectile and melee collision. First player to die loses; no enemy-wave revival between rounds.
- **Online versus: projectiles never hit after first damage** — `GameSession._damagePlayer()` set `player.isInvincible = true` but `Player.update()` only decrements `invincibleTimer` without clearing the boolean. After the first hit both players were permanently invincible. Fixed by clearing `isInvincible` in `_tick()` whenever `invincibleTimer` reaches zero.
- **Online: different biomes per player** — biome type and arena layout were chosen with independent `Math.random()` calls on each client. Fixed by (1) deriving a shared numeric seed from the lobby code, (2) using a deterministic hash of seed + wave number for biome selection, (3) temporarily replacing `Math.random` with a seeded Mulberry32 random generator around `arena.generate()` so both clients produce identical obstacle layouts; wave-1 biome now uses the host's hero type so both clients agree from the start.
- **Online: ghost partner wobbles when moving** — the ghost-player extrapolation multiplied raw input by a hardcoded `4 px/frame` without normalizing the direction vector; diagonal movement overshot by up to 41 % and snapped back at every 50 ms snapshot. Fixed by normalizing the input vector and using the actual hero speed (`stats.speed × speedMultiplier`). Local-player reconciliation dead-zone raised from 10 px to 25 px and blend reduced from 30 % to 8 % per snapshot to eliminate the 20 Hz oscillation caused by RTT drift.
- **Online arena flip / visual corruption**: ghost enemies created via delta snapshot encoding could lack a `radius` field, causing `createRadialGradient` to throw a non-finite error. The unhandled exception skipped `ctx.restore()`, leaving the canvas transform stack corrupted. Progressive corruption across frames caused the arena to visually flip and rotate. Fixed with: (1) early-out guard in `Enemy.draw()` before any canvas operations if `radius` is invalid, (2) safe default `radius`/`color`/`sides` on new ghost enemy objects, (3) `_bodyGradient` cache invalidation when the real radius arrives, (4) `??` instead of `||` in ghost enemy extrapolation (enemy at x=0 was falsy, causing velocity to compound each frame), (5) `ctx.setTransform(identity)` at the start of each game frame to reset any accumulated corruption.

### Fixed
- **Online 2-Player: evil heroes (Green Goblin, Makuta) visible in online hero select**: These heroes are now always filtered out of the online pre-game hero grid, regardless of whether the player has unlocked Evil Mode.
- **Online 2-Player: Story mode shows each player's own hero story**: Both host and guest were independently running story events based on their own hero type, causing out-of-sync narratives. The guest now uses the host's hero type for story event lookups (`window._onlineStoryHero` set in `startOnlineGame`), so both players see the host's hero story.
- **Online Co-op: revive marker appears but game immediately ends**: After a revival marker was placed (setting `player.isDead = true` and `player.hp = 0`), the death check re-fired every frame because `hp` stayed at 0. On the second frame the co-op revival branch was skipped (`!player.isDead` was false) and the death-animation timer started, leading to game over 3 seconds later. Fixed by guarding the outer death check with `!player.isDead`; a separate "both players dead → game over" check now handles the case where both die simultaneously.
- **Online Co-op: ghost enemies appear permanently white on guest side**: When the guest's local projectiles hit ghost enemies (server-synced via snapshot), `enemy.hitFlashTimer = 6` was set on the ghost. Because ghost enemies skip `enemy.update()`, the timer never decremented and the white-flash overlay rendered every frame. Fixed by short-circuiting the collision block for ghost enemies — the projectile is consumed for visual feedback but no damage, flash, or score is applied.
- **Online 2-Player: Play Again after game over starts a new solo game**: Play Again now reopens the online lobby screen so players can rematch without re-entering lobby codes. The button label updates to "🌐 PLAY AGAIN ONLINE" when the session was online.
- **Online 2-Player: Quit Run during online game does not notify partner**: Quitting during an online game now calls `signalGameOver()` first, informing the partner (who will see their own game-over screen), then opens the online lobby for the quitting player.
- **Online 2-Player: Chaos mode available in online despite not being supported in co-op**: The "Chaos" mode button is now hidden in the online pre-game screen. Chaos Shuffle is not supported in any co-op mode (the local co-op gate at `startGame` already blocks it) and was silently broken online.

All four Museum gamepad reads and the GlobalLobbyScene gamepad read now scan for the first device that passes `window.isRealGamepad()` (≥10 buttons, ≥2 axes, connected).
- **Online 2-Player: mode selection screen not appearing / wrong heroes**: `NetworkManager._dispatch()` did not handle the `PRE_GAME` message, so `nm.role` and `nm.lobbyCode` were never set for the Global Lobby invite path (unlike the invite-code path which sets them via `LOBBY_CREATED`/`LOBBY_JOINED`). The server now includes `lobbyCode` in every `PRE_GAME` message (both invite-code and Global Lobby paths).

### Changed
- **Online latency**: Global Lobby player-move updates throttle reduced from 100 ms to 50 ms (doubling update frequency); angle change threshold tightened from 0.09 to 0.05 rad for smoother remote-player rotation. In-game host snapshots increased from every 3 frames (~20 fps) to every 2 frames (~30 fps). Guest flushes input every frame (not every 2) for minimum host-side input lag. Ghost enemies, projectiles, and the host character (player 2 on the guest screen) are now forward-predicted between snapshots using velocity/move-direction, eliminating visible inter-snapshot teleportation on the guest side.

Fixed by adding `!isOnlineGuest` to the death-timer guard so guests never start the local death countdown. Additionally, when the host revives the guest (`isDead` transitions `true → false` in a snapshot), `isPlayerDying` and `playerDeathTimer` are now reset on the guest. - **VersusMenu: controller focus swallowed**: `inputLoop()` did not update `this.inputLoopId` with each new `requestAnimationFrame` call, so `close()` always cancelled a stale handle. When the menu was closed and reopened, two loops ran simultaneously, causing controller input to be processed twice per frame (swallowed or erratic focus). `inputLoopId` is now updated every iteration, and `open()` / `open2PVersus()` cancel any existing RAF before starting a fresh loop.

- **Online lobby WebSocket connection**: When the server hostname was configured via Options → Server (which stores a bare hostname like `192.168.1.x`), the Online lobby passed it raw to `NetworkManager.connect()`, whose `replace(/^http/, 'ws')` found no match and produced a relative URL. Running from a `file://` page, the browser resolved this to `file:///...`, causing "file is not allowed" WebSocket errors. The lobby now routes through `CloudSaveManager._baseUrl()`, which already prepends `http://` and port 3001 for bare hostnames.
